# AI 流式消息抖动测试 (ai-stream-shake-test)

一个用于**复刻并优化 AI 问答「打字机效果」流式渲染抖动**的测试工具。

技术栈：**uni-app + Vue 3 + TypeScript + Sass**，可同时编译到**微信小程序 (mp-weixin)** 与 **H5** 双端，从而在小程序 WebView 环境下真实复现渲染抖动，并对比多种优化方案的效果。

---

## 一、背景：抖动从哪来？

很多 AI 问答的流式渲染**并不是传统的「逐字符队列打字机」**，而是这样一条链路：

```text
后端 SSE delta 分块  →  content 全量累积  →  节流刷新
      →  每次更新对「全量文本」重新做 markdown 解析
      →  重建整棵自定义 token 树  →  原生 view/text 递归重渲染
      →  每帧滚动到底部
```

**打字机观感**来自后端自然分块 + 前端节流；而**抖动 (jitter)** 主要来自以下几点：

1. **每帧全量重解析**：`content` 越长，每帧 `markdown-it` 解析 + 重建 token 树成本越大，导致掉帧、抖动。
2. **大范围响应式更新**：每帧把整个消息数组浅拷贝替换，触发大范围重渲染。
3. **Markdown 半成品跳变**：流式过程中表格 / 列表 / 代码块在「未闭合 → 突然闭合」之间跳变，引起布局跳动（如 `|` 尚未成表 → 突然成表）。
4. **`v-for` 用 index 作 key**：流式结构变化时节点复用错乱。
5. **每帧滚动锚定**：每个 chunk 都触发 scroll，叠加 iOS WebView 的 scroll-anchoring 反向抖动。

## 二、这个项目做了什么

一个极简的**移动端竖屏**测试界面：顶部是**参数设置区**，下方是**对话区**（发任意消息，AI 就把设定好的内容流式逐字返回）。底层流式链路（`markdownParser.ts` / `c-token-render.vue` / `useStreamSim.ts`）刻意还原了上述「全量累积 + 重解析 + 递归重渲染」的实现，因此能真实复现抖动，也便于验证优化的收益。

### 核心思路：生产者–消费者匀速打字机

把**「后端产出」与「前端上屏」解耦**，一次性解决「太快 / 非逐字 / 卡顿」三个问题：

```text
[生产者] 模拟后端 SSE，按 chunk + 延迟投递  →  进入「缓冲区」
                            │
                       (缓冲区削峰填谷)
                            │
[消费者] 前端打字机，以稳定节奏(字/秒)匀速逐字取字上屏
                            │
[渲染]   增量解析(仅重解析末块) + 节流滚动
```

- **解决「太快」**：上屏速度由**打字速度（字/秒）**决定，与后端返回速度无关。后端瞬间吐完也会被缓冲区拦住、匀速上屏。
- **解决「非逐字」**：消费端默认每帧 1 字，形成真正的逐字打字机，而不是「一段段蹦」。
- **解决「卡顿」**：
  1. 渲染改用**增量解析**，只解析最后一个未闭合块，解析耗时与全文长度解耦；
  2. 滚动从「每帧强制 scroll」改为**节流滚动**，消除 scroll-anchoring 抖动；
  3. 缓冲积压时**自动追赶**（单帧多吃几个字但仍平滑），避免长文拖太久。

### 还原真实卡顿

真实后端并非「均匀吐字」，而是**按 SSE 报文 (chunk) 推送**，且每个 chunk 的 `delta.content` 大小极不均匀：既有空包，也有大量 1~2 字小包，偶发一整段 burst；正文中间还会因模型思考 / 调用工具 / 网络波动出现**长停顿**。「chunk 大小不均 + 停顿期」才是卡顿的真正来源。

为此生产者提供两种投递模型：

- **`realistic`（真实卡顿）**：按加权分布产出 chunk（小包为主 + 偶发 burst），到达间隔带较大抖动，并按概率插入 300~1200ms 长停顿；界面会实时提示「后端停顿中」。
- **`smooth`（平滑）**：固定随机 3~12 字 + 平滑延迟，作为对照。

再配合**绕过缓冲**开关，可以直观对比两种上屏方式：

- **开启绕过缓冲**：不做 cps 匀速削峰，按帧尽快逐字吐出缓冲区——burst 到达时「唰」地快速逐字刷出、停顿期则等待，卡顿体现在「忽快忽停」的节奏上（但仍逐字，不整段 pop）。
- **关闭绕过缓冲**：走生产者–消费者匀速削峰，无论后端如何抖动，前端始终平滑逐字。

### 流式 Markdown 消歧（避免符号闪现）

流式逐字接收时，`**加粗**`、`` `代码` ``、`[链接](url)`、`<tag>` 等标记会先冒出「半截符号」再补全，造成符号闪现。项目在上屏前对**末尾未闭合的行内标记做乐观处理**：

- 末尾裸符号（如正在输入的 `*`、`` ` ``、`~~`、`<` / `</` 半截标签）先裁掉，等补全后再显示；
- 已有内容的未闭合标记补一个虚拟闭合，让其立即以样式呈现，符号不露面；
- 正确区分**块级语义**与**行内语义**：无序列表行首的 `* ` bullet 不会被当作斜体定界符；
- 对已闭合文本、比较号 `a < b`、泛型 `arr<T>` 等保持恒等，不误伤、不丢字。

### 逐字上屏动画

给「当前最新出现的那一个字」做一次轻微淡入（仅 `opacity`，不含位移，零布局抖动）。任意时刻至多一个字处于过渡态，随打字自然渐显，不会出现整段半透明；流式结束后自动合并为整段渲染，保证结束态性能。

### 可调参数

| 参数 | 作用 |
| --- | --- |
| **打字速度（字/秒）** | 主控快慢。想看清逐字效果就调小（如 20~30），想快就调大（如 80） |
| **每帧字符数** | 每次上屏字数，`1` 最丝滑；调大更省性能但略跳 |
| **后端到达间隔（ms）** | 模拟后端 SSE 分块到达速度，体现缓冲区削峰能力 |
| **后端投递模型** | `realistic` 还原真实卡顿 / `smooth` 平滑对照 |
| **后端随机抖动** | 到达间隔叠加随机波动，更贴近真实 SSE |
| **积压自动追赶** | 缓冲积压超过阈值时单帧多吃一点，防长文拖太久 |
| **绕过缓冲** | 不经匀速削峰、尽快逐字吐出，还原后端原始卡顿节奏 |
| **缓冲积压** | 实时显示缓冲区里「已到达但还没上屏」的字数 |

## 三、界面说明

竖屏从上到下三块：

- **① 设置区（可折叠）**：数字输入（打字速度 / 每帧字符数 / 后端到达间隔）+ 开关（投递模型 / 随机抖动 / 积压追赶 / 绕过缓冲）+ 缓冲积压与后端状态实时显示 + 模拟回复内容文本框。
- **② 对话区**：用户消息靠右（蓝色气泡），AI 回复靠左（白色气泡），流式逐字上屏。
- **③ 输入栏**：输入框 + 发送键，回车也能发；输出中时发送键禁用。

## 四、运行

```bash
# 安装依赖
npm install

# H5 开发预览
npm run dev:h5

# 微信小程序开发（产物在 dist/dev/mp-weixin，用微信开发者工具打开）
npm run dev:mp-weixin

# 构建
npm run build:h5
npm run build:mp-weixin
```

> 微信小程序端才能真实复现小程序 WebView 的渲染抖动；H5 用于快速预览与逻辑验证。

```text

├── components/
│   ├── c-markdown/          # markdown 渲染入口，按策略切换解析方式

│   ├── markdownParser.ts        # markdown-it 全量解析 + token 树构建

└── pages/index/index.vue    # 测试控制台主界面
```

## 六、优化方案清单

| 方案 | 状态 | 说明 |
| --- | --- | --- |
| 生产者–消费者解耦 | 已落地 | 后端产出进缓冲区，前端按稳定节奏匀速消费上屏，逐字、可控速、不受后端抖动影响（收益最大） |
| 增量解析替代全量重解析 | 已落地 | 只解析「最后一个未闭合块」，已稳定块缓存复用，解析耗时与全文长度解耦 |
| 节流滚动 | 已落地 | 从每帧强制 scroll 改为固定间隔一次，消除 scroll-anchoring 抖动 |
| 流式 Markdown 消歧 | 已落地 | 末尾未闭合标记乐观处理，避免 `**` / `<tag>` 等符号闪现 |
| 逐字淡入动画 | 已落地 | 仅最新字淡入，零布局抖动 |
| 设备分级消费 | 可延伸 | 按设备性能动态调整每帧字符数：低端机多吃字、少更新，高端机切更细 |
| 分块隔离渲染 | 可延伸 | 把 Markdown 按块拆成多个隔离组件，使新增内容不影响已渲染节点，让渲染耗时与长度无关 |
