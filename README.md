# AI 流式消息抖动测试 (ai-stream-shake-test)

一个用于**复刻并优化 AI 问答中「打字机效果」流式渲染抖动**的测试工具。

技术栈与目标业务项目（`lct-ai`）保持一致：**uni-app + Vue 3 + TypeScript + Sass**，可编译到**微信小程序 (mp-weixin)** 与 **H5** 双端，从而在相同环境下真实复现小程序端的渲染抖动，并对比多种优化方案的效果。


---

## 一、背景：抖动从哪来？

对目标项目 AI 流式链路的调研结论：**并不存在传统的「逐字符队列打字机」**（无 per-char queue / setInterval 逐字吐出）。真实实现是：

```
后端 SSE delta 分块  →  content 全量累积  →  80ms 节流刷新(仅小程序)
        →  每次更新对「全量文本」重新 markdown-it 解析
        →  重建整棵自定义 token 树  →  原生 view/text 递归重渲染
        →  每帧 scrollToBottom
```

**打字机观感**来自后端自然分块 + 前端节流；而**抖动 (jitter)** 主要来自：

1. **每帧全量重解析**：`content` 越长，每帧 `md.parse` + 重建 token 树成本越大 → 掉帧/抖动。
2. **列表级整体重建**：每帧把整个消息数组浅拷贝替换，触发大范围响应式更新。
3. **markdown 半成品跳变**：流式中表格/列表/代码块结构在「未闭合 → 突然闭合」间跳变，导致布局跳动（如 `|` 尚未成表 → 突然成表）。
4. **v-for 用 index 作 key**：流式结构变化导致节点复用错乱。
5. **每帧滚动锚定**：每个 chunk 都触发 scroll，叠加 iOS WebView 的 scroll-anchoring 反向抖动。

## 二、这个 demo 做了什么

极简的**移动端竖屏**测试界面：顶部一块**参数设置区**（打字速度 / 每帧字符数 / 后端到达间隔 + 随机抖动 / 积压追赶开关 + 模拟回复内容），下方一块**对话区**（发任意消息 → AI 把设定内容流式逐字返回）。底层流式链路（`markdownParser.ts` / `c-token-render.vue` / `useStreamSim.ts`）对齐真实项目实现。

### 核心优化：生产者-消费者匀速打字机

参考 KM《万字长文丝滑流式输出，打造 ima 小程序极致问答体验》的三级增量方案，把**「后端产出」与「前端上屏」解耦**，解决「太快 / 非逐字 / 卡顿」三个问题：

```
[生产者] 模拟后端 SSE，随机 chunk + 随机延迟 → 投递进「缓冲区」
                              │
                         (缓冲区削峰填谷)
                              │
[消费者] 前端打字机，以稳定节奏(字/秒)匀速逐字取字上屏
                              │
[渲染]   增量解析(仅重解析末块) + 节流滚动(≥120ms)
```

- **解决「太快」**：上屏速度由**打字速度 (字/秒)** 决定，与后端返回速度无关。后端瞬间吐完也会被缓冲区拦住、匀速上屏。
- **解决「非逐字」**：消费端默认每帧 1 字，形成真正的逐字打字机，而不是「一段段蹦」。
- **解决「卡顿」**：① 渲染改用**增量解析**（只解析最后一个未闭合块，耗时与全文长度解耦）；② 滚动从「每帧强制 scroll」改为**节流滚动（≥120ms 一次）**，消除 scroll-anchoring 抖动；③ 缓冲积压时**自动追赶**（多吃几个字但仍平滑），避免长文拖太久。

### 可调参数

| 参数 | 作用 |
| --- | --- |
| **打字速度 (字/秒)** | 主控快慢。想慢一点、看清逐字效果就调小（如 20~30）；想快就调大（如 80） |
| **每帧字符数** | 每次上屏字数，`1` 最丝滑；调大更省性能但略跳 |
| **后端到达间隔 (ms)** | 模拟后端 SSE 分块到达速度，体现缓冲区削峰能力 |
| **后端随机抖动** | 到达间隔叠加随机波动，更贴近真实 SSE |
| **积压自动追赶** | 缓冲积压 >40 字时单帧多吃一点，防长文拖太久 |
| **缓冲积压** | 实时显示缓冲区里「已到达但还没上屏」的字数 |

## 三、界面说明

竖屏从上到下三块：

- **① 设置区（可折叠）**：一行三个数字输入（打字速度 / 每帧字符数 / 后端到达间隔）+ 两个开关（随机抖动 / 积压追赶）+ 缓冲积压实时显示 + 模拟回复内容文本框。
- **② 对话区**：用户消息靠右（蓝色气泡），AI 回复靠左（白色气泡，末尾闪烁光标），流式逐字上屏。
- **③ 输入栏**：输入框 + 发送键，回车也能发；输出中时发送键禁用。

npm install

# 微信小程序开发（产物在 dist/dev/mp-weixin，用微信开发者工具打开）
npm run dev:mp-weixin

npm run build:h5
npm run build:mp-weixin

## 六、优化方案（对齐 KM ima 团队方案，已落地）

参考《万字长文丝滑流式输出，打造 ima 小程序极致问答体验》，核心是三级增量更新，本 demo 已落地其中最关键的几项：

1. **生产者-消费者解耦**（已落地，收益最大）：后端产出进缓冲区，前端按稳定节奏匀速消费上屏 → 逐字、可控速、不受后端抖动影响。
2. **增量解析替代全量重解析**（已落地）：只对「最后一个未闭合块」解析，已稳定块缓存复用，解析耗时与全文长度解耦（文章实测 200ms+ → 5ms 内）。
3. **节流滚动**（已落地）：从每帧强制 scroll 改为 ≥120ms 一次，消除 scroll-anchoring 抖动。
4. **设备分级消费**（可延伸）：文章用 `getDeviceBenchmarkInfo` 按设备性能动态调整每帧 token 数——低端机多吃字少 setData，高端机切更细。可在此基础上把「每帧字符数」做成自动。
5. **组件分块隔离渲染 `<decode/>`**（可延伸）：文章把 markdown 按块拆成多个隔离渲染组件，使新增内容不影响已渲染节点，让渲染耗时变成与内容长度无关的常数。
